EmberKeep — measured performance
================================

Date:        2026-05-02
Hardware:    Intel Core Ultra 7 155H @ 3.80 GHz (Meteor Lake, 16 cores)
             32 GB RAM
             Intel Arc integrated graphics (CPU inference - GPU not used)
OS:          Windows 11 Pro
Model:       Llama-3.2-3B-Instruct, Q4_K_M (~2.0 GB on disk)
Backend:     llama.cpp b8996, CPU only, AVX2 + FMA + F16C
Threads:     6 (pick_threads result on this CPU - leaves headroom for
             Unity main + render + OS)

Throughput
----------
  Tokens / second (story stream): 9.0 tok/s
  Time to first token (cold):     5858 ms  (full storyteller prompt prefill)
  Story length:                   256 tokens in 28364 ms

Frame stability (the 60 FPS claim)
----------------------------------
  Idle (no inference):            243.8 FPS  /  4.1 ms / frame
  During inference (LLM busy):     71.0 FPS  / 14.1 ms / frame
  60 FPS target budget:           16.67 ms / frame
  Per-frame inference budget:      8.00 ms (documented in HUD)

Profiler (Window > Analysis > Profiler, frame during Finn story)
----------------------------------------------------------------
  Main thread:    7.57 ms  (PlayerLoop + UpdateScene)
  Render thread: 10.61 ms  (RenderLoop)
  Total frame:   14.15 ms  (CPU)

Notes
-----
  - The 71 FPS / 14.1 ms result is still 4x faster than the 60 FPS
    target, confirming the worker-thread architecture keeps the render
    loop alive even while the LLM is fully saturating its 6 threads.
  - TTFT is cold-start; in the same session the second call drops to
    ~300-700 ms because the model is already warm in memory.
  - 9 tok/s on CPU is consistent with the "Modern Windows desktop, CPU
    only (8C/16T, AVX2)" sanity-check range in spec section 11.
